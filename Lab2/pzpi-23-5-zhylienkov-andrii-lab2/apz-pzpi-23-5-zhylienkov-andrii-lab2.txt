3 ОПИС ВИКОНАНОЇ РОБОТИ
У межах виконання лабораторної роботи було розроблено мобільний застосунок для операційної системи Android [1], який інтегрується із серверною частиною програмної системи AutoRent, розробленої в попередньому семестрі, та забезпечує взаємодію користувача із функціоналом системи оренди автомобілів. Мобільний застосунок було реалізовано мовою програмування Kotlin [2] із використанням середовища розробки Android Studio. Серверна частина системи була побудована на платформі ASP.NET [3] Core із використанням технології Entity Framework Core [4] для взаємодії з базою даних SQLite [5]. Архітектура створеного рішення базується на клієнт-серверному підході, відповідно до якого мобільний застосунок виконує функції клієнта, а ASP.NET Core-застосунок виконує роль сервера, який обробляє HTTP-запити, взаємодіє з базою даних та повертає результати у форматі JSON.
На початковому етапі виконання роботи було проведено аналіз структури існуючої програмної системи AutoRent, яка вже містила серверну частину, реалізовану за архітектурою ASP.NET Core MVC. У процесі аналізу було досліджено структуру моделей даних, конфігурацію Entity Framework Core, логіку роботи контролерів, а також способи взаємодії із базою даних. Після дослідження існуючої структури було прийнято рішення реалізувати окремий REST API-контролер для забезпечення обміну даними між мобільним застосунком та серверною частиною системи. Для цього у проєкті ASP.NET Core було створено файл MobileApiController.cs (див. додаток Б), який містить HTTP-методи для отримання інформації про автомобілі та створення бронювань. Контролер було реалізовано з використанням атрибутів ApiController та Route, що дозволило організувати REST API відповідно до сучасних стандартів розробки вебсервісів.
У серверній частині було реалізовано метод GET для отримання списку автомобілів із бази даних. Під час виконання запиту сервер звертається до контексту бази даних AppDbContext та виконує отримання колекції автомобілів за допомогою Entity Framework Core. Отримані дані автоматично серіалізуються у формат JSON та передаються мобільному застосунку через HTTP-відповідь. Також було реалізовано метод GET для отримання інформації про окремий автомобіль за його ідентифікатором. Для забезпечення можливості створення бронювання було реалізовано HTTP POST-метод, який приймає дані у форматі JSON, формує новий об’єкт бронювання та зберігає його у базі даних. Для передачі інформації про бронювання було створено окремий клас RentalRequest, який містить поля ідентифікатора автомобіля, імені користувача та дат початку й завершення оренди.
Для забезпечення взаємодії мобільного застосунку із сервером у конфігурацію ASP.NET Core було додано підтримку контролерів та механізм маршрутизації API. Окрім цього, було реалізовано підтримку CORS для усунення можливих проблем із мережевою взаємодією між Android-клієнтом та сервером. У файлі Program.cs було додано політику AllowAll, яка дозволяє виконання HTTP-запитів із мобільного застосунку. Також було змінено конфігурацію launchSettings.json для забезпечення запуску сервера через HTTP-протокол, оскільки Android Emulator за замовчуванням може блокувати роботу із локальними HTTPS-сертифікатами під час розробки та тестування.
Після підготовки серверної частини було розпочато реалізацію мобільного застосунку. Для цього у середовищі Android Studio було створено новий Android-проєкт типу Empty Views Activity із використанням мови Kotlin. Під час створення застосунку було обрано класичний підхід до побудови інтерфейсу користувача за допомогою XML-розмітки та Android Views без використання Jetpack Compose, що дозволило спростити реалізацію та забезпечити стабільну роботу застосунку. Для реалізації мережевої взаємодії із REST API було використано бібліотеку Retrofit [6], яка забезпечує зручне виконання HTTP-запитів та автоматичне перетворення JSON-відповідей у Kotlin-об’єкти. Для підтримки серіалізації та десеріалізації JSON також було підключено бібліотеку Gson Converter.
У структурі мобільного застосунку було створено декілька основних класів. Клас Car.kt було використано як модель даних автомобіля, що містить інформацію про ідентифікатор, марку, модель, рік випуску та вартість оренди за добу. Для опису HTTP-запитів до серверної частини було створено інтерфейс ApiService.kt, який містить метод отримання списку автомобілів через GET-запит до REST API. Для налаштування мережевої взаємодії було реалізовано клас RetrofitClient.kt, у якому створюється об’єкт Retrofit із базовою адресою сервера та конвертером Gson. Під час налаштування мережевої взаємодії було використано спеціальну адресу 10.0.2.2, яка забезпечує доступ Android Emulator до локального сервера, запущеного на комп’ютері розробника.
Для відображення списку автомобілів у мобільному застосунку було використано компонент RecyclerView. У головному вікні застосунку MainActivity було створено RecyclerView та налаштовано LinearLayoutManager для вертикального відображення елементів списку. Для формування окремих елементів списку було реалізовано XML-макет item_car.xml, який містить текстові поля для відображення марки автомобіля, року випуску та вартості оренди. Для роботи із RecyclerView було створено клас CarAdapter.kt, який відповідає за створення елементів списку, прив’язку даних та обробку взаємодії користувача із елементами інтерфейсу. Під час запуску застосунку MainActivity виконує HTTP-запит до серверного API через Retrofit, отримує список автомобілів та передає його в адаптер RecyclerView для відображення на екрані (див. додаток А).
Для реалізації взаємодії користувача із автомобілями було створено додатковий екран DetailActivity, який відкривається після натискання кнопки Details у списку автомобілів. Для передачі інформації між активностями було використано об’єкт Intent, через який передаються дані про вибраний автомобіль. На екрані деталей відображається повна інформація про автомобіль, включаючи марку, модель, рік випуску та вартість оренди. Також на цьому екрані було реалізовано кнопку Book Car, натискання на яку викликає повідомлення про успішне бронювання автомобіля. Таким чином було забезпечено базову взаємодію користувача із функціоналом мобільного застосунку та реалізовано імітацію процесу бронювання автомобіля.
Під час виконання роботи також було створено UML-діаграми, які описують структуру та поведінку розробленої системи. Було побудовано UML-діаграму прецедентів, яка демонструє взаємодію користувача із функціями мобільного застосунку. UML-діаграма компонентів описує архітектуру системи та взаємодію між мобільним застосунком, REST API, серверною логікою та базою даних. Додатково було створено UML-діаграму пакетів, яка демонструє структуру Android-застосунку та взаємозв’язки між основними пакетами й класами, а також UML-діаграму діяльності, що описує послідовність дій користувача та системи під час перегляду та бронювання автомобілів.

13

15          {
16              _context = context;

18  
19          [HttpGet("cars")]
20          public async Task<IActionResult> GetCars()
21          {
22              var cars = await _context.Cars.ToListAsync();
23  
24              return Ok(cars);
25          }
26  
27          [HttpGet("cars/{id}")]
28          public async Task<IActionResult> GetCar(int id)
29          {
30              var car = await _context.Cars.FindAsync(id);
31  
32              if (car == null)
33                  return NotFound();
34  
35              return Ok(car);
36          }
37  
38          [HttpPost("rent")]
39          public async Task<IActionResult> RentCar([FromBody] RentalRequest request)
40          {
41              var rental = new Booking
42              {
43                  CarId = request.CarId,
44                  StartDate = request.StartDate,
45                  EndDate = request.EndDate
46              };
47  
48              _context.Bookings.Add(rental);
49  
50              await _context.SaveChangesAsync();
51  
52              return Ok(new
53              {
54                  message = "Rental created"
55              });
56          }
57      }
